目的: HoldemResources Calculator (HRC) でヘッズアップ(SB=BTN vs BB)プリフロップを
**あなたのアプリの正確な構造**で自前ソルブし、`.json` で書き出す。これを取り込んで
深スタック(13.5〜30BB)のレンジを本物のGTOに置き換え、同時に短スタックの既存レンジを
独立検証する。**自分で解いた出力＝あなたの資産**なので販売アプリに安心して埋め込める。

---

## A. 共通セットアップ(1回だけ)

HRCで HU(2人)スポットを作り、以下を**アプリと一致**させる:

| 項目 | 値 | 備考 |
|---|---|---|
| プレイヤー数 | 2(SB=BTN, BB) | SBが先手・ポストフロップIP |
| スモールブラインド | 0.5 | |
| ビッグブラインド | 1.0 | |
| **アンティ** | **BBアンティ = 1.0**(BBが1BB分を追加) | フルBBアンティ。これが重要(レンジが広くなる) |
| レーキ | 0 | |
| 配当モデル | **まず chipEV**(ICMなし) | アプリの均衡はchipEV基準。検証はchipEV同士で |
| SBオープンサイズ | **2.2×BB**(=2.2に raise) | アプリの `OPEN_SIZE` と一致 |
| アクション | SB: fold / open2.2 / jam、BB: fold / call / 3bet、SB vs 3bet: fold / call / jam | HRCの標準HUツリーでOK。深さで自動的にjam中心になる |

> HRCは設定を保存できます(.hrcv / .json hand config)。テンプレを保存して
> スタックだけ変えて解くと最速です。

各スタック深さ(両者同スタック)で解いて `.json` 書き出し。ファイル名は `hu_chipEV_<BB>bb.json`。

### B-1. 検証用(既存レンジの答え合わせ)— 短スタック push/fold
- **8BB / 10BB / 12BB**
- ここはアプリが「GTO級」と主張している箇所。HRCと一致するか確認する。

### B-2. 本命(深スタックの精度向上)
- **16BB / 18BB / 20BB / 25BB / 30BB**
- ここがアプリの手書き(固定76%オープン等)を置き換える対象。

(余裕があれば 6BB, 14BB も。多いほど補間が滑らかになる)

### B-3. (任意・後日)ICM版
- FT想定のICMノードで chipEV と別に解いて `hu_icm_<BB>bb.json`。

## E. 注意(販売アプリとして)

- **自分でHRCで解いた出力は埋め込み可**(あなたの計算結果)。
- GTO Wizard等**他社の解いたレンジをそのまま再配布するのは規約/知財上NG**になり得る。HRC自前ソルブならこの問題を回避できる。
- ポストフロップ(フロップ以降のGTO助言)は**フェーズ2**。ツールに関わらず別途の大仕事
  (ソルバーはボードごとの解。全ボードは載せられない)。フェーズ1はプリフロップに集中。
